# `scripts/`

Automation for the OpenSIN-overview governance index. Zero runtime dependencies. Node 20+ and Bash 5+ only.

> **Principle:** every file here is either (a) regenerated by CI on a schedule, or (b) invoked on demand during launch week. Nothing in this directory mutates a downstream repo without an explicit `--push` flag or CI context.

## Usage

All Node scripts honour `--help` where it makes sense. All shell scripts do. See individual files for flags.

Common:

```bash
# Full pre-launch sweep (run this before every standup during launch week)
bash scripts/prelaunch-sweep.sh            # fail-fast
bash scripts/prelaunch-sweep.sh --continue # run every step, summarise at the end

# Live dashboard, human-friendly
node scripts/launch-status.js

# Live dashboard, JSON for piping into alerting
node scripts/launch-status.js --json | jq '.results[] | select(.status=="RED")'

# Regenerate the registry (needs `gh auth login`)
npm run regenerate
```

## Contract

- **Idempotent.** Running any script twice with no external state changes produces byte-identical output (except volatile fields like `generated_at`).
- **Exit codes matter.** `0 = ok`, `1 = contract violated`, `2 = bad invocation`. CI reads the exit code, not stdout.
- **Zero deps.** Everything uses Node 20 built-ins (`fs`, `path`, `child_process`, `fetch`, `URL`) and standard POSIX utilities. No `npm install` required to run.
- **gh CLI is optional where plausible.** Scripts that can degrade gracefully without `gh` do so (with a `warn` line); scripts that truly need it (`generate-master-index.js`, `audit-repos.js`) fail fast with a clear message.
- **No secrets in arguments.** Tokens flow via `GH_TOKEN` / `ORG_READ_TOKEN` env vars; never CLI flags.

## Adding a new script

1. Check if an existing script already does 80% of what you need; extend instead of cloning.
2. Follow the shebang + JSDoc header style of existing files. The first block comment IS the help text.
3. Add an entry to the table above AND to `package.json` `scripts` if it should be `npm run …`-able.
4. If it writes into the repo, make sure the output is covered by the linguist-generated rule in `.gitattributes`.
5. Ship with a test that runs in `validate-docs.yml` (even if just `node scripts/your-new-script.js --help | grep -q Usage`).

## What NOT to put here

- Runtime code (belongs in `OpenSIN` / `OpenSIN-Code`).
- Deployment scripts for product surfaces (belong in `OpenSIN-WebApp` / `website-*.opensin.ai`).
- Anything that fetches from user accounts (belongs in `OpenSIN-backend`).
